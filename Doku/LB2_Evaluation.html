  • ✓ Sinnvolle Base-Images: mcr.microsoft.com/dotnet/aspnet:8.0, node:20-alpine, nginx:alpine.
  • ✓ Layer-Caching korrekt: Manifest-First-Pattern (Dockerfile:6–18; frontend/Dockerfile:5–8).
  • ✓ Aggressive Optimierung: --no-install-recommends, apt-Cache-Purge, expliziter curl-Purge (Dockerfile:34, 73–77).
  • ✓ .dockerignore sowohl im Root als auch in frontend/ vorhanden.
  • ✗ Kein USER-Directive in beiden Dockerfiles → Container laufen als root. Das ist genau der Bullet-Point, der die 3. Stufe definiert.
  • ✗ HEALTHCHECK nur in docker-compose.coolify.yml (Zeilen 71–79), nicht im Dockerfile selbst.

Deployment-Ziel 2 / 3

Rubrik: 3 Pkt = Stabil erreichbar mit HTTPS und Healthcheck-gesteuertem Rollout.
  • ✓ Zwei stabile Targets: K8s/Minikube (Ingress NodePort 30080) und Coolify (Port 8090).
  • ✓ Healthchecks auf allen Ebenen:
    • Compose: postgres pg_isready, api bash /dev/tcp (docker-compose.coolify.yml:22–27, 71–79)
    • K8s: livenessProbe, readinessProbe, startupProbe (api-deployment.yaml:89–106)
    • Endpoint: /health in Program.cs:219–222
  • ✓ Healthcheck-Gated Rollout: service_healthy + service_completed_successfully für Migrationen; K8s initContainer Migrator vor API-Pod.
  • ✗ Kein HTTPS: K8s Ingress hat ssl-redirect: "false" (ingress.yaml:7), kein TLS-Block. Coolify-Traefik kann es, ist aber nicht aktiv.
  • ✗ Coolify-Deployment aktuell broken (SSH user-mismatch, siehe Top-Alert).

Quick Win zu 3/3: In Coolify eine Domain hinterlegen + HTTPS-Toggle → Let's Encrypt automatisch. Dauert ca. 5 Minuten.
